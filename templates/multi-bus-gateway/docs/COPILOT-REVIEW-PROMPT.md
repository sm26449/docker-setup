# Copilot review prompt

Paste the block below into GitHub Copilot Chat (Agent / "@workspace" mode, so it
can open files) at the repo root. It asks for an exhaustive, no-flattery review.

---

You are a **senior staff engineer and application-security reviewer**. Perform an
**exhaustive, evidence-based audit** of this repository and give a result worthy
of a paid engagement — specific, prioritized, and honest. Open files as needed;
do not rely on filenames alone. Do **not** be generic or flattering. Cite
`file:line` for every claim.

## What this project is

A self-hosted **general Modbus-TCP/RTU + HTTP → MQTT/InfluxDB gateway** with a
web UI. It began as a single-device **Janitza UMG 512-PRO** power-analyzer
monitor and has been generalized into a **multi-device gateway** with a device
register-map catalog, CSV map import, a **virtual-meter** subsystem (it can also
*act as* a Modbus slave and re-serve live values, e.g. emulate a Carlo Gavazzi
EM24 for a Victron system or a Fronius Smart Meter), and a Template Manager UI.
It runs as a Docker container inside a larger "PV-Stack" home-energy system.

**Backend** (`janitza/`, Python 3.11, FastAPI/uvicorn):
- `api.py` — all HTTP routes (devices, templates, CSV import, virtual meters, status, auth).
- `modbus_client.py` — polling loop, FC3/FC4 read grouping & batching, per-device byte order.
- `register_parser.py` / `encoder.py` — the number engine: byte/word-order matrix
  (`big`/`little`/`badc`/`dcba`) and int16/uint16/int32/uint32/int64/uint64/float/double/string.
- `config.py` — atomic YAML config, `SelectedRegister`, secrets.
- `device_template.py` — JSON template schema + validate/parse; `csv_import.py` — CSV→template.
- `device_templates/*.json` — bundled real maps (Janitza, Eastron SDM630/SDM120,
  Carlo Gavazzi EM24, ABB B23/B21, Schneider iEM3000).
- `virtual_meter.py` — Modbus slave that serves live values on a TCP port.

**Frontend** (`ui/`): `templates/index.html`, `js/app.js` (~8000-line vanilla-JS
SPA, class `JanitzaMonitor`), `css/`, `languages/en.json` + `ro.json` (i18n via a
flat-key `t()` lookup).

**Tests** (`tests/`, pytest, ~216 tests). **Docs** (`docs/`) incl.
`device-catalog.md` (generated by `tools/gen_device_catalog.py`), an audit
(`AUDIT-2026-07-03.md`), and vendor manuals under `docs/vendor/`.

## Invariants you must check are actually upheld

1. **The primary device (Janitza) MQTT/InfluxDB output must be byte-for-byte
   identical** across the generalization — existing history & Home Assistant
   entities depend on the exact topics/fields/values. Verify nothing regressed
   its default `big` byte order or field names.
2. **Bundled device maps must be accurate** — a wrong register address, word
   order, or scale silently corrupts readings (billing/energy data). Assess the
   provenance discipline (each template cites a `source_document`; see
   `docs/device-catalog.md` and `docs/vendor/meters/`). Flag any map you can
   prove wrong.
3. **Alerting sends real SMS** and virtual meters feed control-critical systems —
   treat outward-facing side effects carefully.

## Review dimensions — cover ALL of them

1. **Architecture & design** — layering, cohesion/coupling, the
   source→gateway→sinks model, how cleanly a new device/protocol/sink is added,
   the 8000-line `app.js` maintainability.
2. **Protocol correctness (highest priority — hunt for numeric bugs)** — the full
   byte/word-order matrix in `register_parser.py` + `encoder.py` (are `big`,
   `little`/word-swap, `badc`, `dcba` all correct and mutually inverse between
   parser and encoder?); FC3 vs FC4 read grouping & batching (no cross-type
   merges? gaps handled?); scale-as-divisor correctness incl. 64-bit; signed/
   two's-complement, overflow, partial/short reads, string byte order. Prove any
   bug with a concrete register→bytes→value example.
3. **Device catalog** — template schema robustness, CSV importer resilience
   (weird delimiters/headers/hex), validation coverage, and gaps: no write path
   (FC5/6/15/16), no coils/discrete (FC1/FC2), no SunSpec model library, missing
   common devices/protocols (DNP3, BACnet, M-Bus, Sparkplug B).
4. **Security** — is the API unauthenticated by default over plain HTTP? Secrets
   (`mqtt.password`, `influxdb.token`) stored plaintext in `config.yaml`? Default
   admin creds? Then: input validation on every endpoint, **SSRF** via HTTP
   device sources / json_path fetch, **path traversal** in template/CSV/backup
   upload & export, **XSS** from `innerHTML` in `app.js` (device names, template
   fields, MQTT values rendered unescaped?), CSRF, auth/session handling, ZIP
   backup import (zip-slip).
5. **Reliability & ops** — corrupt-config guard, store-and-forward, watchdogs,
   resource bounds, atomic writes, and **thread safety** (the Modbus client runs
   in threads with an asyncio loop; is shared state / the single socket / config
   access safe under concurrent API writes?).
6. **UI/UX** — SPA structure & state management, the Template Manager + map
   viewer, onboarding, keyboard/accessibility, **i18n parity** (every EN key has
   RO? any hardcoded English strings left?), and render performance (full
   re-render patterns, event-listener leaks).
7. **Testing** — coverage gaps (alerts, event log, backfill, `virtual_meter`,
   security, the number engine's edge cases), and test quality (are they
   asserting behavior or just running?).
8. **Docs & DX** — accuracy vs code, contributor onboarding, the generated
   device-catalog approach.
9. **Performance** — polling efficiency, single socket vs pooling, batching
   limits, SPA payload.

## Deliver

- **Findings**, grouped by severity **Critical / High / Medium / Low**. Each:
  `file:line`, a concrete failure scenario (inputs → wrong/broken output), and a
  specific fix. Put any **correctness bug in the number engine or a device map**
  at the top regardless of other severity.
- **Top 10 improvements**, ranked by impact ÷ effort, each with a one-line
  rationale.
- **Honest overall opinion**: what is genuinely strong, what is weak, and how it
  compares to commercial gateways (Moxa MGate, HMS Anybus, Janitza JUG25).
- **Ratings 1–10** per dimension above, plus an overall score, with justification.
- **A pragmatic 90-day roadmap** (what to fix first and why).

Be concrete, cite code, and tell me what you'd be embarrassed to ship. If a file
is too long, summarize its risky parts rather than skipping it.
